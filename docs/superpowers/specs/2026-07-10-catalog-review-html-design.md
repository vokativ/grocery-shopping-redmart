# RedMart Catalog Review HTML Design

Date: 2026-07-10

## Goal

Improve initial catalog seeding and substantial catalog updates by replacing the Markdown candidate-review step with a temporary, user-friendly HTML review page.

The page should let a non-technical household user approve discovered RedMart products without editing Markdown, YAML, JSON, CSV, or exported files. The agent still owns canonical product resolution and `grocery-catalog.yaml` edits after human approval.

## Current Problem

The current seeding flow writes a `redmart-catalog-candidates-*.md` table as the approval boundary. It is readable for technical users but awkward for normal household review:

- Removing one-offs requires editing Markdown.
- Changing default quantities is not ergonomic.
- Alias review exposes too much structure.
- The approval state is implicit in a file edit rather than a clear action.

## Recommended Approach

Use a temporary static HTML review page generated by the agent.

The page contains the candidate data and all review behavior inline. It is opened locally in the browser. The user reviews the list and clicks one clear approval button, for example:

`Approve 57 products`

On approval, the page stores the final reviewed data in the page state and shows a prominent completion message:

`Approved. You can return to the agent you're using to continue.`

The agent then reads the approved data from the open page through browser control and writes any temporary approval JSON it needs. Only after that does the agent update `grocery-catalog.yaml`.

This avoids requiring a local server, Node.js, a CSV workflow, a download, a copy/export button, or user-visible JSON.

## User Experience

The review page is optimized for laptop and desktop use.

Every discovered candidate starts as included. The user only corrects exceptions.

Each row shows:

- Product title.
- Pack size, observed price, and a short purchase-history hint such as `Bought once`.
- Status pill: usually `Included`; sparing attention tags may be used when needed.
- Editable `Family words` field.
- `Usual quantity` control with plus and minus buttons.
- `Do not include` button.

Rows marked as not included stay in their original position and become visually dimmed. They do not move to a separate section. Their quantity shows as zero, their editing controls are disabled, and the row has an `Include` button to reverse the action.

The page has a clear approval action at the top:

`Approve N products`

For long lists, repeat the approval action at the bottom with plain summary text:

`N products will be remembered. Not-included rows will be skipped.`

The count is the number of included products, not the sum of usual quantities.

## Copy And Labels

Use plain, household-friendly wording.

Use:

- `Review products to remember`
- `Everything is included unless you mark it as not included.`
- `Family words`
- `Family words are optional.`
- `Usual quantity`
- `Do not include`
- `Include`
- `Approve N products`
- `Approved. You can return to the agent you're using to continue.`

Avoid:

- `Remove from catalog` during seeding, because the candidate may not yet be in the catalog.
- `Export`, `Download JSON`, `Copy`, or other implementation-facing actions.
- Product categories or groups unless future user testing shows they reduce confusion.
- Product photos in v1, because collecting them from Lazada orders would add browser work and fragility.

## Attention Tags

Use attention tags sparingly. Most rows should simply be `Included`.

A tag should appear only when it creates a concrete action for the user:

- `Needs words`: the likely household phrase is unclear, usually because there are several related flavors, variants, or brands that need distinct future list terms.
- `Check quantity`: observed quantities are inconsistent, unusually high, or likely not the household's default.
- `Possible duplicate`: the candidate may represent the same household concept as another candidate or an existing catalog item, such as a different pack size or title drift.

If the agent is unsure but the uncertainty does not require user judgment before catalog insertion, record it in the scratch notes instead of adding a tag.

## Data Flow

1. Agent reads RedMart order history using the existing seeding rules.
2. Agent writes a temporary scratch file for browser/session notes.
3. Agent writes a temporary HTML review page from the candidate list.
4. Agent opens the page for the user.
5. User reviews candidates and clicks `Approve N products`.
6. Page marks itself approved and exposes reviewed data in a machine-readable DOM field.
7. Agent reads the approved data from the browser page.
8. Agent writes a temporary approval artifact if useful for traceability.
9. Agent resolves canonical product URLs, item IDs, and SKU IDs as required.
10. Agent updates `grocery-catalog.yaml` only for approved included products.
11. Agent validates YAML parsing and item/SKU collision rules.
12. Agent summarizes the outcome and removes temporary scratch/review artifacts unless the user asks to keep them.

## Approval Boundary

The approval boundary is the user's click on `Approve N products`.

After approval, the agent must not infer additional candidates from the original order list unless the user explicitly asks. The approved page state is the source of truth for:

- Included versus not included.
- Usual quantity.
- Edited family words.
- Any unresolved attention tags.

The agent may still skip an approved product if canonical resolution fails, an item/SKU collision is detected, or the product is unavailable in a way that makes insertion unsafe. These skips must be reported.

## Static Page Mechanics

The page should work as a local static HTML file opened in Chrome.

The page stores review state client-side and, on approval, writes a serialized JSON payload into a hidden or offscreen DOM element that the agent can read through browser control.

The user should never need to interact with the serialized payload.

Fallback behavior:

- If the agent cannot read the approved page state, it may ask the user to keep the page open and retry.
- A manual export/download fallback may exist for debugging, but it should not be visible in the normal UI.

## Generated Approval Payload

The approved payload should include:

- Review schema version.
- Source candidate file/page metadata.
- Approval timestamp.
- Count of included products.
- Candidate records with stable local candidate IDs.
- Product title.
- Pack size.
- Observed price.
- Observed quantity.
- Include status.
- Usual quantity.
- Family words.
- Attention tag, if any.
- Notes needed for downstream canonical resolution.

The payload should not require canonical item IDs or SKU IDs at review time. Those can be resolved after approval using the existing detail/product-resolution pass.

## Error Handling

If the user closes the page before approval, the agent should regenerate or reopen the review page rather than edit `grocery-catalog.yaml`.

If the page is approved with zero included products, the agent should confirm before treating that as intentional.

If a candidate has an attention tag and the user does not change it, the agent may proceed only when the tag is advisory. If the tag blocks safe insertion, the agent should report it instead of guessing.

If Lazada shows a verification challenge during later canonical resolution, the agent must stop and ask the user to clear it, following `AGENTS.md`.

## Testing

Implementation should include:

- A sample candidate dataset generated from current-style candidate rows.
- Static HTML rendering checks for included, not-included, and attention-tag rows.
- Interaction checks for quantity increment/decrement, include/not-include toggling, family-word edits, and approval payload generation.
- A parser/consumer test that reads an approved payload and confirms it preserves include status, usual quantity, and family words.
- Existing catalog validation after any YAML update.

## Out Of Scope For V1

- Editing existing catalog entries directly.
- Removing products already in `grocery-catalog.yaml`.
- Product photos.
- Category/group management.
- Search.
- A persistent web app or server.
- Shopping-cart actions.
- Checkout, delivery-slot, payment, or order placement behavior.
